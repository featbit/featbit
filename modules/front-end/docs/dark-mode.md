# Dark Mode Authoring Guide

This document explains how dark mode works in the FeatBit front-end and
what to do when you add a new component, modal, page, or any styled
element so it looks correct in both themes.

> TL;DR
>
> 1. Use the existing **LESS palette variables** in `src/variables.less`
>    (`@grey5`вҖ“`@grey90`, `@primary-color`, `@green*`, `@red*`,
>    `@purple*`) instead of raw hex values in your component `.less`.
> 2. Run `npm run build:dark-css` after editing any `.less` file.
> 3. Toggle the theme switcher in the user menu and visually check.
> 4. Only fall back to `src/styles-dark.less` when the auto-generator
>    can't reach your case (vendor chrome, brand colors, layout-only
>    overrides, third-party widgets).

---

## How the dark theme is composed

Three stylesheets stack in this order (later wins):

| # | File | Source | Purpose |
|---|------|--------|---------|
| 1 | `styles-dark-vendor.css`   | generated from ng-zorro's compiled dark theme by `scripts/build-dark-vendor-css.mjs` | All ng-zorro defaults flipped for dark, prefixed with `html[data-theme="dark"]`. The Ant blue accent is rewritten to FeatBit green at build time. |
| 2 | `styles-dark-overrides.css` | generated by `scripts/build-dark-overrides.mjs` from every `*.component.less` and `src/styles*.less` | Auto-emits dark equivalents of any **grayscale** color literal it finds (e.g. `#fff` вҶ’ `#262626`, `#f1f1f1` вҶ’ `#333333`). Brand greens / reds / purples are left untouched. |
| 3 | `src/styles-dark.less`      | hand-written | Final tweaks: app chrome (header/menu), brand swaps, edge-cases the generator can't reach (e.g. multi-mode `nz-select` centering), and rules that need to win on specificity. |

`angular.json` wires all three into every build configuration. The two
generated files are gitignored вҖ” they are regenerated by the
`postinstall` and `build:dark-css` npm scripts.

**Docker images** all run `npm install --ignore-scripts` and then rely on
`npm run build:prod` (which chains `build:dark-css`) to generate the dark
stylesheets exactly once during the image build. This avoids generating
the CSS twice (once during install via `postinstall`, again during build),
and keeps `modules/front-end/Dockerfile`, `modules/front-end/wss.Dockerfile`,
and `docker/ui-api-bundle/Dockerfile` consistent. Locally, `npm install`
still runs `postinstall` so the dev server has dark CSS available
immediately.

## Rules of thumb when writing component `.less`

- **Prefer the LESS palette variables**. Examples from `variables.less`:
  - Surfaces: `@grey5`, `@grey10`, `@grey20` (light tints), `@grey90`
    (near-black). The auto-generator knows these literal values and
    rewrites them.
  - Text: `@grey80` for body, `@grey60` for secondary, `@grey40` for
    helper text.
  - Brand: `@primary-color` (`#3CC798`), `@green50-color` (`#5CE6B8`),
    `@green70-color` (`#23AD7F`), `@red50-color` (`#FF513A`),
    `@purple50-color` (`#6751D6`).
- **Avoid pure black `#000` and pure white `#fff` for body chrome.**
  Use `@grey5`/`@grey10` (light) or rely on the dark palette in
  `styles-dark.less` (`@dk-page`, `@dk-surface`, вҖҰ) when you do need to
  hand-tune.
- **`!important` is rarely needed in component scope.** Component
  styles are encapsulated by Angular; the auto-generator emits dark
  rules with `!important` to win over your light rule when
  `data-theme="dark"`. If you also use `!important`, you may force the
  generator's dark rule to lose. Only use `!important` if the cascade
  genuinely demands it.
- **Use `nzColor` on `<nz-tag>`** when you want a semantic accent (the
  vendor handles both themes). Custom tag classes need explicit
  hand-tuning in `styles-dark.less`.

## Patterns the generator handles automatically

The `build-dark-overrides.mjs` script compiles every component LESS,
walks the resulting CSS with PostCSS, and rewrites color tokens whose
**HSL saturation is low** (max-min RGB delta вүӨ 35) вҖ” i.e. anything that
reads as gray. The mapping depends on the **role** of the property
(background, border, text, shadow) and the source lightness:

| Role        | Source lightness `l`        | Dark output                         |
|-------------|-----------------------------|-------------------------------------|
| background  | `l вүҘ 0.99` (white-ish)      | `#262626` (page)                    |
| background  | `l вүҘ 0.92`                  | `#333333` (container surface)       |
| background  | `l вүҘ 0.85`                  | `#3d3d3d` (hover / divider)         |
| background  | `l вүҘ 0.75`                  | `#474747` (raised pill / chip)      |
| background  | `l < 0.75`                  | _(left as-is вҖ” already dark enough)_ |
| border      | `l вүҘ 0.92`                  | `#363636`                           |
| border      | `l вүҘ 0.80`                  | `#3d3d3d`                           |
| border      | `l вүҘ 0.65`                  | `#454545`                           |
| border      | `l < 0.65`                  | _(left as-is)_                      |
| text/color  | `l вүӨ 0.20`                  | `rgba(255, 255, 255, 0.92)`         |
| text/color  | `l вүӨ 0.40`                  | `rgba(255, 255, 255, 0.85)`         |
| text/color  | `l вүӨ 0.55`                  | `rgba(255, 255, 255, 0.65)`         |
| text/color  | `l вүӨ 0.70`                  | `rgba(255, 255, 255, 0.45)`         |
| text/color  | `l > 0.70`                  | _(left as-is вҖ” already light)_      |

So `#fff`, `#fafafa`, `#f1f1f1`, `#e6e6e6`, `#d4d4d4`, `#717D8A`, etc.
all flip automatically. Saturated colors вҖ” brand greens (`#3CC798`,
`#5CE6B8`), red (`#FF513A`), purple (`#6751D6`), the chart palette вҖ”
have a high RGB delta and pass through untouched.

Source alpha is preserved: `rgba(0, 0, 0, 0.45)` becomes
`rgba(38, 38, 38, 0.45)`, keeping subtle shadows subtle.

## When you need to hand-tune (`styles-dark.less`)

Open `src/styles-dark.less` and add an override under the relevant
section. Use the palette variables defined at the top of the file.

Common reasons you'll need a hand-tuned rule:

1. **A brand-colored element doesn't read well on dark.**
   Example: the secondary purple button вҖ” the dark text `#6751D6` was
   too dim against the page; we lightened the foreground to `#B5A6FF`
   and added a translucent purple background.
2. **A vendor rule wins on specificity over the auto-generated rule.**
   The auto-generator emits `html[data-theme="dark"] .x { вҖҰ }` (one
   class). Vendor rules sometimes nest two or three classes deep. Add
   a more specific selector (matching the vendor's ancestors) to win.
3. **Multi-element widgets (selects, date pickers, modals) need
   layout-level fixes**, not just colors. Example: `nzMode="tags"`
   selectors needed flex-centering on the placeholder; layout was
   never a color concern, so the generator couldn't help.
4. **You want a different surface tier** than the generator chose.
   Example: the project list on Get Started has three nesting tiers
   (page / project card / env entry). The generator collapses tiers
   2 and 3 to the same `#333`. We restore separation in
   `styles-dark.less` using `@dk-tier-2` / `@dk-tier-3`.
5. **You're swapping non-CSS assets** (e.g. logos with dark fills).
   Use `content: url('assets/...-dark.svg')` on the `<img>` so the
   markup doesn't need to change.

### Specificity gotcha

The auto-generator may emit:

```less
html[data-theme="dark"] .body-container .ant-tag { background: #333 !important; }
```

That has specificity `(0,0,3,1)`. A naive override of:

```less
html[data-theme="dark"] turn-flag-on .ant-tag { background: green !important; }
```

is `(0,0,2,2)` вҖ” it loses, even with `!important`, because higher
class count wins inside the same `!important` tier. Match the ancestor
chain (e.g. include `.body-container`) to bump your specificity.

## Worked example: adding a new component

Say you're adding an `AuditFilterCardComponent` вҖ” a card with a title, a
filter dropdown, a help link, and a status pill.

### 1. Generate

```bash
ng generate component features/safe/audit-logs/audit-filter-card
```

### 2. Author the HTML вҖ” no theme awareness needed

```html
<div class="filter-card">
  <h3 class="filter-card-title">Filter audit events</h3>
  <nz-select class="nz-select-36" [(ngModel)]="selectedType">
    <nz-option nzValue="all" nzLabel="All events"></nz-option>
    <nz-option nzValue="flag" nzLabel="Flag changes"></nz-option>
  </nz-select>
  <a class="help-link" routerLink="/help/audit">Learn more</a>
  <nz-tag class="status" [class.on]="enabled" [class.off]="!enabled">
    {{ enabled ? 'On' : 'Off' }}
  </nz-tag>
</div>
```

No `[ngStyle]`, no `[class.dark]`, no theme-service injection. Just normal
Angular.

### 3. Author the LESS вҖ” use the palette variables

```less
@import "variables";

.filter-card {
  background: #ffffff;                // page-relative surface вҶ’ #262626 in dark
  border: 1px solid @grey10-color;    // #EAECEE вҶ’ #3d3d3d in dark
  border-radius: 8px;
  padding: 16px;

  .filter-card-title {
    color: @grey80-color;             // dark slate вҶ’ rgba(255,255,255,0.85)
    font-size: 14px;
    font-weight: 600;
    margin-bottom: 12px;
  }

  .help-link {
    color: @primary-color;            // brand green вҖ” passes through untouched
    &:hover { color: @green70-color; }
  }

  nz-select { width: 100%; margin-bottom: 8px; }
}
```

Note: **prefer scoped, specific class names** (`.filter-card-title`,
not just `.title`) so the auto-generator's emitted rule stays scoped to
your component and doesn't bleed across the app.

### 4. Regenerate the dark stylesheet

```bash
npm run build:dark-css
```

(`build:prod` and `build:dev` chain this automatically, so a normal
build covers you.)

> вҡ пёҸ **`npm start` does NOT chain `build:dark-css`.** `start:en` / `start:zh`
> call `ng serve` directly, and Angular's HMR has no knowledge of the Node
> generator script. **After editing any `.less` file during dev, re-run
> `npm run build:dark-css` in another terminal and refresh the page** вҖ” or
> the auto-overrides will be stale and dark mode will silently misrepresent
> your component. Tracked as a follow-up to wrap `ng serve` in a chokidar
> watcher.

### 5. Test both themes

Start the dev server, open the page in light mode, then click the theme
switcher in the user menu. You should see:

- Card surface flips white вҶ’ `#262626`.
- Border flips to a subtle dark border.
- Title flips dark slate вҶ’ light text at 85% white.
- The brand-green help link and the `nz-tag.on/off` pill look identical
  in both modes (covered by globals in `styles-dark.less`).

### 6. The 5% case where you'd crack open `styles-dark.less`

You'd only hand-tune if:

- **A vendor / auto-gen rule wins over you.** Inspect in DevTools, find
  the winning rule, then add a more specific override:
  ```less
  html[data-theme="dark"] .filter-card .ant-select-selector {
    background-color: @dk-page !important;
  }
  ```
- **You picked a non-grayscale color that doesn't read well on dark.**
  (e.g. a chart accent against `#262626`.)
- **You introduced a new nesting tier** the generator can't reason
  about (your card sits inside another card and they collapse to the
  same `#333`).
- **You used a generic class name** with a hardcoded bg вҖ” the
  generator emits an unscoped rule that bleeds. Either rename the
  class, scope the original rule, or add a transparent override.

### Mental model

> Write the component in light mode using palette variables. Run the
> dark-css build. Verify in dark mode. **95% of the time you're done;
> 5% of the time you add one block to `styles-dark.less`.**

## Workflow for adding a new component

1. **Write the component using palette variables.** No raw hex unless
   you genuinely need a brand color.
2. **Build:** `npm run build:dark-css` (regenerates the auto-overrides),
   then `npm start` (remember: `start` does not auto-rebuild dark CSS вҖ”
   re-run `build:dark-css` after every `.less` edit) or `npm run build:prod`.
3. **Toggle the theme** via the user-menu theme switcher (or set
   `localStorage.setItem('featbit:theme', 'dark')` and reload).
4. **Inspect every state**: hover, focus, selected, disabled, error,
   loading, empty.
5. **If something looks wrong**, identify why:
   - Is the offending color non-grayscale? вҶ’ add a `styles-dark.less`
     entry that picks a dark-friendly shade.
   - Is a vendor rule winning? вҶ’ add a more specific override.
   - Did your component use `!important` in the light rule and now
     the dark generator loses? вҶ’ drop the `!important` from the light
     rule, or duplicate the override in `styles-dark.less`.
6. **Re-run the build** and re-test both themes.

---

## Testing checklist

- [ ] Element renders correctly in **light** mode.
- [ ] Element renders correctly in **dark** mode (no white flashes,
      no invisible borders/text).
- [ ] **All interactive states** (hover, focus, active, selected,
      disabled) have sufficient contrast in dark mode.
- [ ] **Switching themes** at runtime updates the element without a
      reload.
- [ ] If the component embeds Monaco/code editor: the editor theme
      follows (`ThemeService` syncs this).
- [ ] No console errors after running `npm run build:prod`.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---------|--------------|-----|
| White background "blinks" through the component for a frame on theme toggle | Component sets a hardcoded white bg in inline styles or `[ngStyle]` | Move the color into LESS; use `transparent` in inline style. |
| Border invisible on a dark surface | Border color was a near-black light-mode shade (e.g. `#d4d4d4`) that auto-mapped to nearly-page color | Use `@grey20`/`@grey30` in light, then add `border-color: @dk-border-medium` (or `@dk-border-bright`) in `styles-dark.less`. |
| Text too dark to read | Foreground color is brand purple/blue/red; dark generator skipped it | Add an override in `styles-dark.less` lightening the text (e.g. tints derived from the brand color). |
| Tag/badge doesn't have its semantic color | Vendor `.ant-tag` rule wins | Use `nzColor="green"` etc. on `<nz-tag>` if available, else add a higher-specificity override. |
| Multi-mode `nz-select` placeholder/cursor mis-aligned | Vendor flex layout differs in dark; only `.ant-select-selection-search-input` is content-anchored | Apply `transform: translateY(-50%)` to the **placeholder only**, never the search input. |
| Logo/SVG looks black on dark | Asset has dark fills baked in | Add a `-dark.svg` variant; swap via `content: url(...)` in `styles-dark.less`. |
| Generic class name (e.g. `.tip`, `.title`, `.label`) gets a wrong dark bg in places that didn't define one in light | The auto-generator scopes its output by what was written in the source LESS. A bare `.tip { background: #f1f1f1 }` in one component generates an unscoped `.tip { background: #333 !important }` that bleeds into every `.tip` on the site | Two options: (1) scope the original light rule (e.g. `section.foo .tip { вҖҰ }`) so the generator emits a scoped dark rule; or (2) override with `transparent` in `styles-dark.less` for the affected ancestors. |
| Specificity battle between auto-gen and your hand-tuned override | Auto-gen rule includes the full ancestor chain from the source LESS; your one-class override doesn't match its specificity | Match the ancestor chain in `styles-dark.less` (e.g. include `.body-container` if the auto-gen rule does). |

## Files quick-reference

| Path | Purpose |
|------|---------|
| `src/variables.less` | The single source of truth for the light palette and brand colors. |
| `src/styles-common.less` | Light-mode global styles. |
| `src/styles-dark.less` | Hand-tuned dark overrides (this file's companion). |
| `src/app/core/services/theme.service.ts` | Light/dark/system preference, applies `data-theme` attribute, syncs Monaco theme, persists in `localStorage`. |
| `src/app/core/components/theme-switcher/` | Icon-only toggle in the user menu. |
| `scripts/build-dark-vendor-css.mjs` | Regenerates `styles-dark-vendor.css` from ng-zorro. Re-run when ng-zorro is upgraded. |
| `scripts/build-dark-overrides.mjs`  | Regenerates `styles-dark-overrides.css` from current component LESS. Re-run on every LESS edit. |
| `package.json` `build:dark-css` | Runs both generators. Also wired into `postinstall`. |
