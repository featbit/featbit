Algorithm review notes:

- Bayesian and bandit analysis are implemented as JSON builders inside both provider-specific `ReleaseDecisionExperimentService.cs` files; there is no separate algorithm service yet.
- The stats query is part of the analysis contract. The algorithm never reads raw events directly; it consumes per-variant aggregates produced by `IExperimentStatsService`.
- Binary metrics use proportion-style input `{ n, k }`. Continuous metrics use moment-style input `{ n, sum, sum_squares }`.
- Run analysis now carries the run sampling contract into stats through `RunId`, `AssignmentUnitSelector`, `LayerKey`, `SliceStart`, `SliceEnd`, `LayerTrafficPercent`, and `AnalysisSamplingPlan`. Feature flag evaluation decides the actual served variation; layer eligibility only decides whether an exposure can participate in the run; analysis sampling decides how much of each actual variation enters evidence.
- The layer hash key is `layerKey + assignmentUnit`. The sampling hash key is `flagKey + actualVariation + assignmentUnit`, so E2E preset sampling is stable across repeated runs while still sampling inside the actual served variation. These hashes do not reinterpret or override the FeatBit feature flag traffic split.
- SRM is checked from observed users per analyzed variant using a chi-square survival function. With sampling plans, the observed set is actual exposure after layer eligibility and per-variation include-rate sampling, not the full feature-flag rollout.
- Bayesian output is decision-support output, not an automatic run decision write. The method persists `AnalysisResult`, but does not populate `Decision`, `DecisionSummary`, or learning fields.
- Bandit output recommends traffic weights, but the method does not apply those weights back to a FeatBit flag or run audience settings.
- `ForceFresh` exists on the request contract but is not used by the current Postgres implementation.
- Mongo now implements the same release-decision run analysis loop at the service level, using Mongo stats aggregation and the same Bayesian/bandit helper logic as the Postgres path.

### Query mechanism

- First exposure per `user_key` in window determines the user's variant only on the legacy direct stats path without a run sampling plan.
- Metric events are counted only when `occurred_at >= exposure_ts`.
- Aggregation returns per-variant users/conversions/value moments.
- Binary metrics force `once`; continuous metrics can use `count`, `sum`, or `average`.
- Postgres implements this with SQL CTEs; Mongo implements the same contract in memory after loading matching exposure and metric documents.
- When `AnalysisSamplingPlan` is supplied, stats use actual exposure `variation_id` as the served variation. The query resolves an assignment unit from `AssignmentUnitSelector`, applies optional layer eligibility, then samples inside each actual variation using the entry's `includeRate`.
- Custom assignment-unit selectors are strict. If the requested property is missing from exposure or metric event properties, that event is excluded; stats never silently fall back to `user_key` for a custom selector.
- Duplicate exposure events are expected. After eligibility and sampling filters, the first included exposure per assignment unit wins. Duplicate metric events are also expected and are aggregated per assignment unit using `once`, `count`, `sum`, or `average`.
- Postgres and Mongo persist computed assignment snapshots for audit/debug. ClickHouse applies the same query semantics without adding a ClickHouse assignment table.

### Query mechanism

- `OLAPProvider` is independent from `DbProvider`. Core experiment workspace data still reads/writes through Postgres or MongoDB.
- `OLAPProvider=ClickHouse` overrides only `IFeatureFlagInsightsService`, `IFeatureFlagEndUserStatsService`, and `IExperimentStatsService`.
- ClickHouse experiment stats applies the same current run sampling semantics as Postgres and Mongo: actual exposure `variation_id` is the served variation, optional layer eligibility gates assignment units, and per-variation `analysisSamplingPlan` controls whether a served assignment enters evidence. It does not add a ClickHouse assignment-snapshot table; `release_decision_run_assignments` remains a Postgres/Mongo audit/debug store.
- In ClickHouse mode, event ingestion is `evaluation-server -> Kafka featbit-insights -> ClickHouse Kafka table -> materialized views -> typed MergeTree tables`.
- The back-end API does not consume or write `Topics.Insights` for the ClickHouse event path; it only queries ClickHouse.
- Missing `OLAPProvider` falls back to the selected `DbProvider`, preserving standard Postgres/Mongo behavior.

### ClickHouse script review

- Database: `featbit`.
- Kafka queue table: `featbit.kafka_insight_events_queue`, consuming `featbit-insights` as `JSONEachRow`.
- Exposure table: `featbit.release_decision_exposure_events`, partitioned by `(env_id, toYYYYMM(exposed_at))` and ordered by `(env_id, flag_key, exposed_at, cityHash64(user_key))`.
- Metric table: `featbit.release_decision_metric_events`, partitioned by `(env_id, toYYYYMM(occurred_at))` and ordered by `(env_id, event_name, occurred_at, cityHash64(user_key))`.
- Materialized views split insight events by event type: `FlagValue` rows become exposure events, and other rows become metric events.
- Timestamp conversion uses `fromUnixTimestamp64Micro(...)`, matching the current insight message timestamp payload.
