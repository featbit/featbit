## 2. Release decision A/B test algorithm

The actual server-side algorithm code is concentrated in one backend file.

| Area | Files | What changed |
|---|---|---|
| Server algorithm | `modules/back-end/src/Infrastructure/Services/EntityFrameworkCore/ReleaseDecisionExperimentService.cs` | `AnalyzeRunAsync` fetches stats, builds primary/guardrail input data, and chooses Bayesian vs bandit output. Contains Bayesian comparison, relative/absolute effect, probability, credible interval, risk, SRM check, normal distribution helpers, posterior estimates, and bandit weight computation. |
| Application trigger | `modules/back-end/src/Application/ReleaseDecisions/AnalyzeReleaseDecisionExperimentRun.cs` | Thin request handler that delegates to `IReleaseDecisionExperimentService.AnalyzeRunAsync`. |
| API trigger | `modules/back-end/src/Api/Controllers/ReleaseDecisionExperimentController.cs` | Adds `POST /runs/{runId}/analyze`. |
| MCP trigger | `modules/back-end/src/Api/Mcp/ReleaseDecisionMcpTools.cs` | Exposes `featbit_release_decision_analyze_run` for agent-driven analysis. |
| Frontend trigger/display | `modules/release-decision-web/src/components/experiment/experiment-run-table.tsx`, `analysis-markdown.tsx`, `experiment-run-traffic-config.tsx`, `traffic-pool-view.tsx`, `stage-content-panel.tsx` | Lets users select Bayesian A/B/n or bandit, configure control/treatments/arms, run analysis, and render Bayesian or bandit results. Rendering only; calculation is server-side. |
| Comparison script | `modules/release-decision-web/scripts/compare-bandit-py-ts.ts` | Added for bandit comparison/verification support in the web module. |

Important split: PostgreSQL has the real `AnalyzeRunAsync` implementation. Mongo `ReleaseDecisionExperimentService.AnalyzeRunAsync` only records "analyze requested" and returns the experiment, so Mongo is not algorithm-parity with Postgres in this PR.

## Quick review notes

- The main runtime truth for release-decision experiment data is now FeatBit API/PostgreSQL, not Prisma tables in `release-decision-web`.
- `modules/release-decision-web/prisma/schema.prisma` still maps old-looking table names such as `experiment`, `experiment_run`, `activity`, and `message`. It appears to be used for generated TypeScript types, not runtime data, but it is worth checking before anyone regenerates Prisma against the new DB schema.
- Mongo support is present for event queries and basic experiment records, but run creation/update/analyze is not equivalent to the PostgreSQL implementation.
- Feature flag insights now depend on release-decision exposure ingestion. If no `FlagValue` events reach `release_decision_exposure_events`, both old flag insight screens and new A/B stats will look empty.
